### INSTRUCCIONES DE ANÁLISIS:

1. VISIÓN HOLÍSTICA Y DOMINIO (OBLIGATORIO)
Analiza la selección cruzándola estrictamente con el contexto del usuario.

Si existe contexto:
- Identifica el dominio implícito.
- Traduce ese dominio en problemas técnicos concretos (ej. concurrencia, consistencia, latencia, sincronización, etc.).
- Está prohibido mencionar el dominio sin derivar implicaciones técnicas reales.

---

2. GESTIÓN DE CONTEXTO LIMITADO (CRÍTICO)
Si no hay contexto suficiente:
- Prohibido inventar casos de uso.
- El análisis debe centrarse exclusivamente en:
  - interacción entre tecnologías
  - modelo de ejecución
  - acoplamiento
  - implicaciones arquitectónicas

---

3. INTERACCIÓN ENTRE TECNOLOGÍAS (OBLIGATORIO)
Se debe de reflejar la interacción entre tecnologías del stack.

---

4. CAPACIDADES NATIVAS VS EXTENSIONES (OBLIGATORIO)
- No atribuyas capacidades que no sean nativas.
- Si algo depende de una librería:
  - debes indicarlo explícitamente
  - debes mencionar la librería o mecanismo 
- Usa expresiones como:
  "requiere integrar...", "se suele implementar con...", "depende de..."

---

5. CONCRECIÓN TÉCNICA (OBLIGATORIO)
Cada punto del DAFO debe incluir:
- un mecanismo técnico concreto (ej. event loop, pool de conexiones, serialización, locking…)
- y una consecuencia directa (ej. latencia, consumo de memoria, bloqueo de hilos…)

Ejemplo válido:
"provoca bloqueo del event loop bajo operaciones síncronas intensivas"

Ejemplo inválido:
"mejora el rendimiento"

---

6. REALISMO OPERACIONAL (OBLIGATORIO)
Debes analizar comportamiento en condiciones reales:

- carga concurrente
- consumo de CPU/memoria
- número de conexiones
- latencia de red
- sincronización entre procesos

Está prohibido omitir estos aspectos.

---

7. PROHIBIDO FEATURE LISTING
Está prohibido reformular características como fortalezas.

Ejemplo prohibido:
"incluye ORM → facilita desarrollo"

Solo se permiten consecuencias técnicas reales derivadas del uso.

---

8. PROHIBIDO CONTENIDO GENÉRICO
Está prohibido incluir:
- "gran comunidad"
- "es escalable" (sin justificar cómo)
- "ecosistema maduro"
- "Docker / Kubernetes / CI/CD" (si no es estrictamente necesario)
- cualquier afirmación válida para cualquier stack

---

9. PRIORIZACIÓN
El análisis debe centrarse en:
- cuellos de botella reales
- fricciones entre componentes
- impacto bajo carga
- trade-offs arquitectónicos

---
